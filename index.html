Primele Variații de Hashing

Hashing-ul a evoluat din necesitatea de a verifica integritatea datelor și de a stoca parolele în mod sigur.
Primele funcții hash erau simple și vulnerabile — astăzi sunt fundația criptografiei moderne.

DEPRECAT · 1953

CRC (Cyclic Redundancy Check)

Prima formă de verificare a integrității datelor. Nu este o funcție hash criptografică, ci un checksum folosit pentru detectarea erorilor de transmisie.

Output: 16 sau 32 biți
Scop: Detectare erori, nu securitate
Vulnerabilitate: Ușor de manipulat deliberat
Folosit azi: Rețele, stocare (nu criptografie)
DEPRECAT · 1991

MD4 (Message Digest 4)

Creat de Ron Rivest, a fost prima funcție hash criptografică populară. Rapid, dar cu coliziuni descoperite rapid de Dobbertin (1995).

Output: 128 biți (32 hex chars)
Runde: 3 runde de procesare
Vulnerabilitate: Coliziuni în minute pe PC modern
Descendenți: MD5, SHA-1, RIPEMD
DEPRECAT · 1992

MD5 (Message Digest 5)

Îmbunătățire a MD4, a dominat internetul timp de un deceniu. Folosit masiv pentru parole și verificare fișiere. Complet compromis din 2004.

Output: 128 biți
Runde: 4 runde × 16 operații
Vulnerabilitate: Coliziuni în secunde (Flame malware 2012)
NU folosi pentru: Parole, semnături digitale
SLAB · 1995

SHA-0 & SHA-1

Dezvoltat de NSA, SHA-1 a înlocuit MD5 pentru un deceniu. Coliziunile practice au fost demonstrate de Google (SHAttered, 2017) cu 9 quintilioane de operații.

Output: 160 biți (40 hex chars)
Runde: 80 runde de procesare
Vulnerabilitate: Collision attack demonstrat 2017
Retras: Certificate TLS din 2017
LIMITAT · 1992

RIPEMD-160

Creat în Europa ca alternativă la SHA. Folosit în Bitcoin și PGP. Nu a fost spart, dar are output mic față de standardele moderne.

Output: 160 biți
Design: Două pipeline-uri paralele
Status: Sigur, dar lent și output mic
Folosit în: Bitcoin (P2PKH addresses)
CONTEXT · 1976

DES & Hash-urile bazate pe cifruri

Înainte de hash-uri dedicate, criptografii foloseau cifruri de bloc (DES) în moduri speciale pentru a construi funcții hash unidirecționale.

Metodă: Construcția Matyas-Meyer-Oseas
Problemă: Lent, cheie fixă
Lecție: Hash-urile dedicate sunt mai eficiente
Moștenire: Principii preluate în SHA-1

Algoritmii de Hashing

O funcție hash criptografică transformă date de orice dimensiune într-un output de dimensiune fixă.
Proprietăți cheie: unidirecționalitate, rezistență la coliziuni, efect avalanșă (1 bit diferit → output complet diferit).

STANDARD · SHA-2

SHA-256 / SHA-512

Familia SHA-2, creată de NSA în 2001. SHA-256 produce 256 biți prin 64 de runde de compresie. Fundament al Bitcoin și TLS 1.3.

Structură: Merkle–Damgård construction
Runde: 64 (SHA-256) / 80 (SHA-512)
Avalanșă: ~50% biți schimbați
Folosit în: Bitcoin, TLS, JWT, S/MIME
MODERN · SHA-3

SHA-3 / Keccak

Câștigătorul competiției NIST 2012. Singurul hash cu design Sponge Construction — complet diferit de SHA-2, imun la atacuri de extensie de lungime.

Structură: Sponge (Absorb + Squeeze)
State intern: 1600 biți
Permutare: Keccak-f[1600], 24 runde
Variante: SHA3-224/256/384/512
RAPID · BLAKE3

BLAKE3

Cel mai rapid hash criptografic securizat (2020). Paralelizabil, de 10× mai rapid decât SHA-256. Bazat pe ChaCha20 și Merkle tree intern.

Throughput: ~10 GB/s pe CPU modern
Paralelism: SIMD + multi-thread
Output: 256 biți (extensibil)
Folosit în: Rust std, ZFS, Bao
PAROLE · BCRYPT

bcrypt / scrypt / Argon2

Hash-uri adaptative pentru parole. Intentionat lente și intensive în memorie pentru a bloca atacurile brute-force și dictionary attacks cu GPU.

bcrypt: Cost factor configurabil (2^N)
scrypt: N, r, p — memorie intensivă
Argon2: Câștigător PHC 2015, recomandat acum
Argon2id: Protecție side-channel + GPU
AUTENTIFICARE · HMAC

HMAC (Hash-based MAC)

Combină o funcție hash cu o cheie secretă pentru autentificarea mesajelor. HMAC-SHA256 garantează atât integritatea cât și autenticitatea datelor.

Formula: H(K⊕opad ‖ H(K⊕ipad ‖ m))
Cheie: Orice lungime (recomandată ≥ 32B)
Folosit în: JWT, OAuth 2.0, API keys
Rezistă la: Length extension attacks
STRUCTURI · MERKLE

Merkle Tree Hashing

Structură de hash-uri arborescente unde fiecare nod conține hash-ul copiilor săi. Permite verificarea eficientă a datelor mari (blockchain, Git, TLS).

Rădăcină: Hash al întregului set de date
Verificare: O(log n) — fără descărcare totală
Folosit în: Bitcoin, Ethereum, Git, IPFS
Proprietate: Orice modificare schimbă rădăcina

Securitate P2P

Comunicația peer-to-peer ridică provocări unice: nu există server central de încredere.
Handshake-ul TLS, schimbul de chei și criptarea de capăt-la-capăt sunt esențiale.

// TLS 1.3 Handshake — Pas cu Pas

PASUL 01 · CLIENT
ClientHello
Clientul trimite: versiunea TLS suportată, lista de cipher suites (ex: TLS_AES_256_GCM_SHA384), extensii (SNI, ALPN) și un număr aleator de 32 bytes (client_random). De la TLS 1.3, clientul trimite și key_share (cheia publică Diffie-Hellman) direct.
PASUL 02 · SERVER
ServerHello + Certificate
Serverul răspunde cu cipher suite ales, server_random și propria cheie publică DH (key_share). Trimite certificatul digital (X.509) și o semnătură digitală pentru autentificare. Calculează imediat master secret folosind ECDH.
PASUL 03 · AMBII
Key Derivation (HKDF)
Ambele părți calculează independent același shared_secret prin ECDH. Din acesta, HKDF (HMAC-based Key Derivation Function) derivă cheile de sesiune: client_write_key, server_write_key, IV-uri pentru AES-GCM.
PASUL 04 · FINAL
Finished + Date Criptate
Ambele trimit un mesaj "Finished" autentificat cu HMAC pentru a confirma că handshake-ul nu a fost manipulat. De acum toate datele sunt criptate cu AES-256-GCM sau ChaCha20-Poly1305.

// Metode de Schimb al Cheii de Criptare

CLASIC · DH

Diffie-Hellman (DH)

Primul protocol public de schimb de chei (1976). Alice și Bob ajung la același secret fără să îl transmită prin rețea. Baza tuturor protocoalelor moderne.

Bază matematică: Problema logaritmului discret
Formula: K = g^(ab) mod p
Problemă: Fără Perfect Forward Secrecy dacă cheile sunt statice
Vulnerabil la: MITM fără autentificare
MODERN · ECDH

ECDH (Elliptic Curve DH)

Versiunea pe curbe eliptice a Diffie-Hellman. Chei de 256 biți oferă securitate echivalentă cu RSA-3072, dar sunt mult mai rapide.

Curbă standard: Curve25519 (X25519)
Alternativă: P-256 (NIST), secp256k1 (Bitcoin)
Folosit în: TLS 1.3, Signal, WhatsApp
PFS: ✅ cu chei efemere (ECDHE)
ASIMETRIC · RSA

RSA Key Exchange

Clientul criptează premaster secret cu cheia publică a serverului. Simplu dar fără Perfect Forward Secrecy — compromiterea cheii private dezvăluie toate sesiunile trecute.

Dimensiune cheie: 2048-4096 biți recomandat
Problemă: NU are Perfect Forward Secrecy
Status TLS: Eliminat din TLS 1.3
Totuși util: Semnături digitale, certificate
VIITOR · PQC

Post-Quantum Cryptography

Calculatoarele cuantice pot sparge RSA și ECDH prin algoritmul Shor. NIST a standardizat ML-KEM (Kyber) și ML-DSA (Dilithium) în 2024.

ML-KEM (Kyber): Key encapsulation
ML-DSA (Dilithium): Semnături digitale
Bază matematică: Lattice-based cryptography
Hibrid: X25519 + Kyber (recomandat acum)

// Metode Adiționale de Securizare P2P

IDENTITATE · PKI

Certificate & PKI

Infrastructura de chei publice (PKI) cu Certificate Authorities (CA) rezolvă problema autentificării în P2P. Web of Trust (PGP) este alternativa descentralizată.

X.509: Standard certificate TLS
CT Logs: Certificate Transparency — auditabil public
DANE: DNS-based Authentication (no CA needed)
P2P alternativ: PGP Web of Trust
MESAGERIE · E2E

Signal Protocol (Double Ratchet)

Protocolul folosit de Signal, WhatsApp, Messenger. Combină X3DH pentru schimb inițial cu Double Ratchet Algorithm pentru Perfect Forward Secrecy la fiecare mesaj.

X3DH: Extended Triple DH — schimb inițial
Double Ratchet: Cheie nouă la fiecare mesaj
Proprietăți: PFS + Break-in Recovery
Open source: ✅ Auditat public
INTEGRITATE · PINNING

Certificate / Key Pinning

Aplicația memorează (pinuiește) cheia publică sau certificatul expected. Orice CA compromis nu poate emite certificate false pentru domeniu. Apără MITM attacks.

HTTP Public Key Pinning: Deprecat (risc lockout)
Alternativă: Expect-CT header + CT Logs
Mobile: Pinning în codul aplicației
HSTS Preloading: HTTPS forțat în browsere
MODERN · NOISE

Noise Protocol Framework

Framework folosit de WireGuard, Lightning Network, WhatsApp transport. Definește modele de handshake criptografic clar și auditabil, fără complexitatea TLS.

WireGuard: Noise_IKpsk2 pattern
Proprietăți: Mutual auth, PFS, Identity hiding
Avantaj: Suprafață mică de atac
Primitiv: Curve25519 + ChaCha20 + BLAKE2
SESIUNE · SESSION KEYS

Chei de Sesiune Efemere

Fiecare sesiune P2P generează chei noi, care sunt șterse după terminare. Chiar dacă un atacator înregistrează traficul acum, nu-l poate decripta mai târziu.

Perfect Forward Secrecy: Cheie nouă / sesiune
Generare: CSPRNG (ChaCha20, /dev/urandom)
Durata de viată: Sesiune sau interval scurt
Ștergere sigură: Zeroize în memorie
AUTENTIF. · MUTUAL TLS

mTLS (Mutual TLS)

Ambele părți prezintă certificate — nu doar serverul. Esențial în arhitecturi zero-trust, microservicii și rețele P2P unde ambii peers trebuie autentificați.

Standard TLS: Doar serverul are certificat
mTLS: Client + Server — autentificare reciprocă
Folosit în: Kubernetes, Istio service mesh, P2P
Avantaj: Zero implicit trust, fără parole
